**30 分钟 PPT 大纲（建议 12–14 页 + Q&A）**
- 目标受众：硬核同事（关心架构、可靠性、成本、落地）
- 核心主张：Skills = “可移植的过程化专业知识包”，用“渐进式加载”解决上下文与一致性问题；MCP 提供访问，Skills 提供方法；Tools 是固定能力，Skills 是按需加载的知识与流程

## Slide 1 — 标题：Agent Skills：把“我们的方法”装进可复用的能力包（1:30）
**页面要点**
- Skills 解决：重复讲需求、上下文浪费、流程不一致、交付不可控
- 一句话：Skills = 轻量开放格式的“工作方法与流程包”，按需加载

**讲稿（口播）**
今天我不讲“模型有多强”，我讲“怎么把我们团队的工作方式变成可复用、可评估、可组合的能力”。Skills 的定位很明确：不是替代工具，不是连外部系统，而是把“遇到某类任务应该怎么做”固化成标准化的知识与流程，让通用 agent 在不同场景下更稳定地产出一致结果。

---

## Slide 2 — 痛点：没有 Skills 时，成本如何失控（2:00）
**页面要点**
- 每次都要重述：指令、模板、参考资料、边界条件
- 上下文污染：越堆越长，召回不稳定，越“提示工程”越脆
- 交付不一致：同一任务多次运行输出风格/结构漂移

**讲稿**
硬核同学都见过：你写了一套“超长提示词”，第一次效果不错，第二次换个输入就跑偏；或者团队里每个人有自己的 prompt 版本，没人能保证一致性。更关键的是上下文窗口是有限资源：把所有规则都塞进提示词，本质是在烧 token 并赌模型记住关键点。Skills 的价值在于把知识分层、把流程外置、把执行变得更确定。

---

## Slide 3 — 定义：Skills 是什么（2:00）
**页面要点**
- Skill = 一个文件夹：指令、脚本、模板、资源（轻量开放格式）
- 目标：给通用 agent 增加“领域 expertise + 可重复 workflow + 新能力”
- 开放与可移植：跨不同 skills 兼容 agent 复用（规范驱动）

**讲稿**
把 Skill 当作“可发现、可加载、可执行”的能力单元：它不是一个大而全的系统，而是围绕某类任务的操作指南与资源集合。关键点有三个：第一，它把知识组织成文件；第二，它按需加载，不把所有内容塞进上下文；第三，它可组合，可以把多个 Skill 拼成复杂工作流。

---

## Slide 4 — 范式转变：从“专用 Agent”到“通用 Agent + Skills”（2:00）
**页面要点**
- 过去：Research/Coding/Marketing 等“专用 agent”各自脚手架与提示
- 现在：通用 agent 以“代码/文件系统”为通用接口
- Skills 提供：过程化知识、团队/公司上下文、可重复流程

**讲稿**
以前我们的直觉是“为每种岗位造一个专用 agent”。问题是：维护成本高、上下文重复、能力不通用。新的范式是通用 agent + 简单脚手架（文件系统与脚本执行），再通过 Skills 注入不同领域的做事方法。这其实更像软件工程：能力模块化、可组合、可测试。

---

## Slide 5 — 核心架构：Progressive Disclosure（三层渐进式加载）（3:00）
**页面要点**
- Level 1 Metadata：启动时加载，约 ~100 tokens/skill（发现与路由）
- Level 2 Instructions：命中请求时加载（建议 <5k tokens）
- Level 3 Resources：执行时按需读取/运行（几乎无限，文件不直接进上下文）

**建议图示**
- 三层堆叠图：Metadata → Instructions → Resources（标注加载时机与成本）

**讲稿**
这是 Skills 体系里最硬核也最关键的设计：渐进式加载。Metadata 让 agent 在“很低 token 成本”下知道有哪些技能可用；Instructions 只在相关时加载，避免无关内容污染；Resources 走文件系统读取与脚本执行——注意，文件本身不进上下文，只有读取/执行的输出进上下文。这是从“提示工程”走向“上下文工程”的核心。

---

## Slide 6 — Skill 结构：SKILL.md 的两部分（2:00）
**页面要点**
- 顶部 YAML Frontmatter：name、description（必填）+ 可选字段
- 正文 Markdown：工作流、输入输出、边界情况、示例
- 可选目录：/references、/scripts、/assets（按需）

**建议图示**
- 目录树示例（SKILL.md + references/ + scripts/ + assets/）

**讲稿**
实现层面一个 Skill 最小就是 SKILL.md：上面是元数据，下面是指令正文。元数据负责“让系统找到你”；正文负责“让 agent 正确使用你”。更复杂的逻辑放在 resources：比如 reference 文档、脚本、模板。这种结构让 Skill 既能轻量，也能逐步复杂化。

---

## Slide 7 — 编写规范（硬约束 + 最佳实践）（2:30）
**页面要点（来自材料的关键约束）**
- name：≤64 字符；小写字母/数字/连字符；不能以连字符开头结尾；需与父目录同名；推荐动名词 -ing
- description：≤1024 字符；不能为空；要写清“做什么/何时用”，包含可触发关键词
- 正文建议：控制在 500 行内；详细资料拆到 references；路径统一用正斜杠

**讲稿**
如果你的 name/description 写得不具备“可检索性”，这个 Skill 等于不存在。硬核的点在于：description 不是 marketing 文案，而是路由条件的一部分——要让 agent 更容易判断“何时该加载”。另外正文不要无限膨胀，超过一定规模就拆 references；路径用正斜杠是为了跨环境一致。

---

## Slide 8 — Degrees of Freedom：控制“自由度”来控制可靠性（2:00）
**页面要点**
- 高自由度：文字方向，适合创意输出（容许多做法）
- 中自由度：给模式/伪代码/示例（推荐但可变）
- 低自由度：指定脚本与顺序（强约束、最稳定）

**讲稿**
这张是落地时的“可靠性旋钮”。如果你要稳定交付，把关键步骤做成低自由度：直接要求执行 scripts 并按顺序跑；如果是创意类工作，用高自由度避免过度约束。团队落地时要先决定每类任务的自由度目标，否则你会在“写得很长但不稳定”和“写得很死但不通用”之间来回震荡。

---

## Slide 9 — 执行环境与安全边界：为什么 Skills 更“工程化”（2:00)
**页面要点**
- 通过文件系统读写与脚本执行完成确定性任务
- 安全限制（材料明确）：无网络、不能运行时安装包、CPU/内存限制
- 含义：更可控、更可复现，但要用内置能力与本地资源设计流程

**讲稿**
Skills 的一大优势是“可控”：很多复杂操作不靠模型脑补，而靠脚本处理。与此同时环境也更受限：没有网络、不能临时 pip install。对工程落地来说这不是缺点，而是约束带来的可复现性——你能明确知道依赖是什么、输出怎么来的、失败如何定位。

---

## Slide 10 — 可组合：从单技能到工作流（2:00）
**页面要点**
- Skills 可组合：品牌规范 Skill + PPT Skill + 数据分析 Skill
- 多 Skill 协作：agent 自动决定顺序、通过文件共享中间产物
- 示例（材料）：分析销售数据 → 生成图表 → 生成汇报 PPT

**讲稿**
单个 Skill 像一个库函数；多个 Skill 组合就像一个 pipeline。关键在“共享介质”：不是把所有中间结果塞进对话，而是落文件，再按需读取。这也是为什么 Resources 层几乎无限：你可以把大数据/大图表/大模板放在文件系统里，让上下文保持干净。

---

## Slide 11 — 边界 1：Skills vs Tools（2:00）
**页面要点**
- Tools：固定能力集合，定义常驻上下文（参数/描述）
- Skills：动态加载的知识与流程（按需进入上下文）
- “tools on demand”：Skill 里可以带 scripts，把新能力封装成可执行资源

**讲稿**
Tools 更像运行时的系统调用：你有就有，没有就没有；并且 tool 定义常驻上下文。Skills 则是按需加载的“工作方法”。更有意思的是 Skills 可以携带脚本，相当于把部分能力做成“随用随取”的工具扩展——这让能力扩展更工程化，也更接近我们熟悉的模块化开发。

---

## Slide 12 — 边界 2：Skills vs MCP（2:00）
**页面要点**
- MCP：连接外部系统/数据（数据库、API、服务）= access
- Skills：告诉 agent 如何使用数据计算指标/做决策 = expertise
- 组合模式：MCP 拉数据 → Skill 定义口径/方法 → scripts 做计算与验证

**讲稿**
一句话：MCP 给你“能拿到什么”，Skill 给你“拿到以后该怎么做”。硬核同学会关心：如果只上 MCP，不上 Skills，你得到的是原始数据但没有口径；如果只上 Skills，不上 MCP，你有方法但没有数据。最优解是二者组合：接入与方法分离，职责清晰。

---

## Slide 13 — 边界 3：Skills vs Subagents（1:30）
**页面要点**
- Subagent：独立上下文与权限，适合并行/隔离任务
- Skills：可增强主 agent 与所有 subagents 的方法论与流程
- 组合方式：主 agent 分配任务给 subagents；subagents 共享同一套 Skills 标准

**讲稿**
Subagents 是“组织方式”，Skills 是“知识与流程”。你可以让多个 subagent 并行干活，但要让他们产出一致，就需要共享同一套 Skills。否则你会得到并行的混乱，而不是并行的效率。

---

## Slide 14 — 评估与质量：让 Skill 可测（2:30）
**页面要点**
- 单元测试思路：skills / queries / files / expected_behavior
- 成功标准要可判定：能读文件、提取目标、按模板输出、可编译/可执行
- 更进一步：人工反馈 + 覆盖计划用到的所有模型

**讲稿**
硬核团队一定要把 Skill 当“可测试资产”。不要只写“应该更好”，要写可判定的 expected behavior：是否正确读取输入文件、是否覆盖所有学习目标、是否严格遵循输出模板、产物是否能编译或被下游工具消费。否则 Skill 永远停留在 demo 阶段，上不了生产。

---

## Slide 15 — 落地路线图 + Takeaways（2:30 + Q&A）
**页面要点**
- 选 3 类高频任务做 Skill：模板化强、口径一致、ROI 高
- 设定自由度策略：关键路径低自由度（脚本化），创意路径高自由度
- 建立“Skill Registry”：命名规范、评审门槛、测试用例、版本管理
- 三句 Takeaway：
  1) Skills 是上下文工程：分层加载，控制 token 与污染  
  2) MCP 解决 access，Skills 解决 expertise  
  3) 可靠性来自脚本化 + 可评估标准

**讲稿**
落地建议很直接：先挑高频、强模板、强口径的任务做 Skills，这类 ROI 最大；同时把关键步骤脚本化，减少模型自由发挥；最后把 Skills 纳入工程治理：评审、测试、版本、发布。你会发现这和我们治理代码库非常像——因为本质上 Skills 就是“面向 agent 的工程资产”。  
（引导 Q&A）大家可以从三个方向提问：成本（token/维护）、可靠性（怎么测、怎么防跑偏）、组织（怎么共享、怎么治理）。

# 额外：给你准备的“硬核 Q&A 弹药”（可放到备份页）
- 为什么 Progressive Disclosure 不是“噱头”？  
  - 因为它直接约束 token 成本、降低上下文干扰，并把大内容转移到文件系统/脚本输出上，形成可控信息通道。
- Skills 会不会变成另一种 prompt 维护地狱？  
  - 会，如果不做：规模控制（<500 行）、references 拆分、自由度分级、测试与版本治理。
- 如何判断该用 MCP 还是 Skill？  
  - 需要连外部系统/实时数据：MCP；需要标准口径/流程/模板/决策规则：Skill；通常两者组合。
- 可靠性怎么提升到“可上线”？  
  - 关键路径低自由度（脚本+固定顺序）+ 可判定 expected_behavior + 多模型测试 + 人工回归样例集。
